1. RESUMEN EJECUTIVO
--------------------------------------------------------------------------------
Se resolvieron tres problemas encadenados que mantenían la aplicación en
producción inutilizable para el flujo de Cálculo U — Techo:

  (A) Freeze de UI al seleccionar el elemento "Techo" en Zona D.
  (B) TypeError en minificado: "k.zonas.join is not a function".
  (C) Desconexión silenciosa de la integración GitHub→Vercel desde el
      15-abril (4 días) por un archivo untracked que rompía el build.

Los tres problemas se diagnosticaron, corrigieron y verificaron en
producción. La aplicación vuelve a operar y el flujo Techo → Zona D
calcula correcciones sin bloquear la interfaz.

2. DIAGNÓSTICO — CADENA DE CAUSAS
--------------------------------------------------------------------------------

2.1 Freeze en techo (síntoma visible)
    · El motor `generarCorrecciones` en src/data.js ejecutaba una búsqueda
      lineal síncrona con paso de 10 mm dentro de la función `_findMinEsp`.
    · Para techos en Zona D (U_max = 0,38 W/m²·K), el rango de espesores
      útiles (hasta 300 mm × 6 estrategias × N alternativas de aislante)
      generaba miles de iteraciones sin ceder el hilo.
    · El event loop quedaba bloqueado 10–30 s sin repintar — el usuario lo
      percibía como "la app se colgó".

2.2 Cascada de doble cálculo en React (síntoma agravante)
    · `calcularConCapas` (PanelCalcU) llamaba a `onCalcUChange` en el
      padre, que hacía `setCalcUInit(...)`.
    · Esto cambiaba el prop `initData`, lo que disparaba
      `useEffect([initData])` en PanelCalcU y relanzaba `calcGlaser` +
      `generarCorrecciones` redundantemente.
    · Resultado: cada interacción ejecutaba el motor DOS veces.

2.3 TypeError "k.zonas.join is not a function"
    · Aparecía en consola sobre el bundle `index-DQEFUQ1m.js`.
    · Origen: algunas filas del catálogo `SC` (data.js) traían `zonas` como
      undefined o con tipo inesperado tras ciertas rutas de importación,
      mientras el código llamaba `.join()` asumiendo array.
    · Agravante: al ser una ruta minificada, el stack no era auditable a
      primera vista.

2.4 Desconexión Vercel ↔ GitHub (root cause de por qué NADA llegaba)
    · El archivo src/modules/AdminTokens.jsx existía en el disco local,
      era importado desde App.jsx (línea 22), pero nunca se había
      commiteado. Estaba como "?? src/modules/AdminTokens.jsx" en
      `git status`.
    · Cada push posterior al 15-abril disparaba un build en Vercel, pero
      Rollup/Vite fallaba con:
          "Could not resolve ./modules/AdminTokens.jsx from src/App.jsx"
          "Command 'npm run build' exited with 1"
    · Vercel marcaba el deploy como Error y el CDN seguía sirviendo el
      último bundle exitoso: `index-DQEFUQ1m.js` del 15-abr (commit
      14bcd2f).
    · Evidencia forense (curl a producción antes del fix):
          Age: 304003            (~3,5 días)
          Last-Modified: Thu, 16 Apr 2026
          X-Vercel-Cache: HIT
    · Esto explicaba por qué los commits df1ee93 y 3efdf6c, aunque
      correctos, no producían cambios visibles — nunca llegaron a
      desplegarse.

4.1 src/data.js — Motor de correcciones asíncrono con yield real
    ----------------------------------------------------------------------
    Antes: loop síncrono bloqueante.
    Ahora (líneas ~971–1002):

        const _YIELD = () => new Promise(resolve => setTimeout(resolve, 0));
        const PASO_GRUESO = 40;
        const MAX_ITER = 50;

        async function _findMinEsp(minEsp, maxEsp, tryFn) {
          let iter = 0;
          let espesorTest = minEsp;
          let prevFail = null, firstPass = null;

          // Fase 1: salto grueso (40 mm) hasta pasar el umbral
          while (espesorTest <= maxEsp && iter < MAX_ITER) {
            iter++;
            if (iter % 5 === 0) {
              await new Promise(resolve => setTimeout(resolve, 0));
            }
            if (tryFn(espesorTest)) { firstPass = espesorTest; break; }
            prevFail = espesorTest;
            espesorTest += PASO_GRUESO;
          }
          if (firstPass === null) return null;

          // Fase 2: refinación fina (10 mm) entre prevFail y firstPass
          const fineStart = prevFail === null ? minEsp : prevFail + 10;
          for (let e = fineStart; e < firstPass && iter < MAX_ITER; e += 10) {
            iter++;
            if (iter % 5 === 0) {
              await new Promise(resolve => setTimeout(resolve, 0));
            }
            if (tryFn(e)) return e;
          }
          return firstPass;
        }

    Justificación técnica:
    · `setTimeout(0)` es una **macrotarea** (no microtarea como
      Promise.resolve). Solo las macrotareas ceden el hilo al browser
      para repaint y permiten que el spinner de "Calculando..." se
      vea. `Promise.resolve()` agota la cola de microtareas antes del
      repaint y no libera la UI.
    · Paso grueso 40 mm + refinación fina 10 mm reduce iteraciones en
      ~75% respecto a un barrido uniforme de 10 mm.
    · Hard-cap MAX_ITER=50 garantiza terminación acotada incluso con
      parámetros extremos.
    · `iter % 5` yield cada 5 iteraciones: balance entre fluidez visual
      y overhead de cambio de contexto.

    Callers migrados (todos con `await`):
      - C1 (aumentar aislante principal)     ~línea 1042
      - C2 (agregar aislante adicional)      ~línea 1064
      - C3 (reemplazar aislante)              ~línea 1093
      - C4 (aumentar capa interior)           ~línea 1119
    Además se insertó `await _YIELD()` entre estrategias C1–C6 y dentro
    de los loops `for (const alt of AISLS)` en C1/C2/C3/C6.

6. RIESGOS RESIDUALES Y RECOMENDACIONES
--------------------------------------------------------------------------------

6.1 Tiempos del motor
    · En máquinas muy lentas, techos con gran número de alternativas de
      aislante aún pueden tardar 2–4 s. El spinner ahora SÍ se ve (era
      el fix principal), pero conviene medir percentil 95 en campo.
    · Recomendación: si se observa >5 s, añadir una LRU cache para los
      resultados de `generarCorrecciones` con llave `(elem,zona,U_req,
      composición)`. Ya está contemplado en el diseño.

6.2 Mantenimiento del catálogo SC
    · El blindaje `|| ''` / `|| []` protege contra datos mal formados,
      pero conviene añadir un test unitario que valide en build time
      que todas las filas de SC tengan `zonas: string` y `usos: array`.
    · Sugerencia: un script npm `validate:catalog` ejecutado en pre-push.

6.3 Integración Vercel ↔ GitHub
    · El incidente raíz (AdminTokens.jsx untracked 4 días) habría sido
      detectable con:
        (a) CI check sobre `git status --porcelain` en pre-push hook.
        (b) Notificación de Vercel a email cuando un build falla en
            producción (Settings → Notifications).
    · Recomendado: activar ambos.

6.4 Badge de versión
    · Actualmente es una cadena estática en App.jsx. Para automatizar,
      inyectar `VITE_BUILD_HASH` con `git rev-parse --short HEAD` en un
      script de build, y leerlo con `import.meta.env.VITE_BUILD_HASH`.

6.5 Archivos AUDITORIA_*.txt
    · Los archivos AUDITORIA_NORMACHECK.txt, AUDITORIA_SISTEMA_
      CONSTRUCTIVO.txt y AUDITORIA_FREEZE_TECHO_FIX.txt siguen
      untracked. Si quieres que formen parte del repo, agrégalos; si
      no, conviene añadirlos a .gitignore para no verlos en git status.
